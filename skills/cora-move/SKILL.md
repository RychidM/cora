# Vault Move

Renames or relocates a project/module and fixes up everything that points
at it. The vault uses Obsidian `[[wikilinks]]` and several index files, so
a move is only safe if all references move with it.

---

## Step 1 — Locate the vault and the source

Resolve vault root (see `cora-logger` Step 1).

Resolve the source project folder (same as the promoter):
`projects/{name}/` first, then `projects/*/{name}/` for modules. Use the
first match. Note whether it's top-level or a module, and whether it has
modules of its own. If it can't be found, stop and tell the user.

---

## Step 2 — Determine the operation and destination

| the user wants | From → To |
|----------|-----------|
| rename | `projects/{old}/` → `projects/{new}/` (or `projects/{parent}/{old}/` → `projects/{parent}/{new}/`) |
| move module to another parent | `projects/{a}/{mod}/` → `projects/{b}/{mod}/` |
| promote module to top-level | `projects/{parent}/{mod}/` → `projects/{mod}/` |
| demote top-level to module | `projects/{name}/` → `projects/{parent}/{name}/` |

New names are kebab-case. Confirm the resolved source and destination
paths with the user before moving — this changes many files.

Refuse / stop if:
- the destination already exists (never overwrite),
- a target parent doesn't exist,
- the move would nest deeper than one level
  (`projects/{parent}/{module}/` is the max — no `parent/child/grandchild`),
- the source has its own modules and the user is demoting it to a module
  (that would create illegal two-level nesting) — flag and ask.

---

## Step 3 — Move the folder

Move the entire source folder (and all its contents and any modules) to
the destination path, preserving everything verbatim.

If available tooling can't move a directory (no shell move), give the user the
exact `mv`/source→dest paths and stop — don't half-apply by copying some
files.

---

## Step 4 — Fix up references

Update every place that names or links the project. Use
uniquely-anchored edits.

1. **`projects/_INDEX.md`** — update the project's row: its name cell and
   its `[[{path}/OVERVIEW]]` link. If it became/stopped being a module,
   re-nest the row (`&nbsp;&nbsp;↳ {name}` under the parent, or flatten).
2. **`AGENTS.md`** — same update in the Active Projects table.
3. **The moved `OVERVIEW.md`** — if it's now a module, ensure the
   "System Context" block points at the correct parent `[[../OVERVIEW]]`,
   and set its `parent:` frontmatter to the new parent's name; if it's no
   longer a module (promoted to top-level), remove that block and clear
   `parent:` back to empty. On rename, update any self-referential
   title/`project:` frontmatter.
4. **Parent `OVERVIEW.md`(s) — `submodules:` frontmatter and ACTIVITY.md**:
   - If the source had a parent, remove the module's (old) name from the
     old parent's `submodules:` list.
   - If the destination has a parent, add the module's (new) name to the
     new parent's `submodules:` list (every top-level project already
     has an `ACTIVITY.md` from the template, so there's no feed to
     create here — just the list entry).
   - If a top-level project is **demoted** to a module, delete its own
     `ACTIVITY.md` (it no longer rolls up anything itself).
   - If a module is **promoted** to top-level, it has no `ACTIVITY.md`
     (modules never do) — create one from the skeleton (see
     `cora-logger`'s ACTIVITY.md skeleton) so it's ready once it gains
     modules of its own.
   - Also fix any prose references to the module in the old parent's
     `OVERVIEW.md`, and add to the new parent's if cross-module decisions
     mention it.
5. **`[[wikilinks]]` across the vault** — search all `*.md` for links to
   the old path or name (`[[{old}/OVERVIEW]]`, `[[{old}]]`, etc.) and
   repoint them to the new path/name. Use `cora-find` semantics to catch
   them; show the user the list before rewriting if there are many.
6. **`.project-paths`** — the synced-repo registry keys on the project;
   update the entry if the project name is part of it.

---

## Step 5 — Flag stale synced files

A move changes `# Module Context` / `# System Context` headers and parent
links, so any repo agent files (CLAUDE.md, GEMINI.md, …) generated by
`cora-project-sync` are now stale. Tell the user to re-run `/cora-sync` for
the affected repo(s).

---

## Step 6 — Report

```
Moved {old path} → {new path}
  Updated: _INDEX.md, AGENTS.md, OVERVIEW.md (incl. parent:/submodules: frontmatter), {N} wikilinks, .project-paths
  Re-run /cora-sync for: {repo(s)} — agent files are now stale
```

If any reference couldn't be updated, list which and why — don't leave
dangling links silently.

---

## Rules

- **Never overwrite** — refuse if the destination exists
- **One level of nesting max** — `projects/{parent}/{module}/`, no deeper
- **Move, don't copy-and-delete** — and never destroy content
- **Fix every reference** — index rows, AGENTS.md, parent OVERVIEWs,
  wikilinks, `.project-paths`; a move that leaves dangling links is a
  failed move
- **Keep `parent:`/`submodules:` frontmatter in sync on both sides** —
  a move that updates the folder but not these fields will get flagged
  by `cora-lint`'s Check 7
- **Confirm before moving** — show resolved source/destination paths and
  the wikilink rewrite list first
- **Flag stale synced files** — synced repos need a re-sync after a move
